# OpenEdifier Agent 指南

本文件适用于整个仓库。子目录如有更具体的 `AGENTS.md`，以更深层文件为准。

## 项目目标

OpenEdifier 是一个本地优先的 EDIFIER 音箱控制 monorepo，提供跨平台 Rust SDK、CLI、原生 macOS MVP，以及规划中的 iOS、公开语言绑定和 Home Assistant 集成。

当前唯一经过实机验证的型号是固件 `01.00.00` 的 S260。不要把同族协议、其他型号的公开项目或厂商应用中的字段直接视为 OpenEdifier 已支持能力。

## 目录职责

- `crates/open-edifier-core`：与型号无关的公共类型、错误和同步契约。
- `crates/open-edifier-aaec`：可复用的 AA EC/BB EC 分帧，不包含型号命令语义。
- `crates/open-edifier-discovery`：mDNS 发现和设备身份识别，不拥有控制端口。
- `crates/open-edifier-s260`：S260 的 JSON 控制、状态解析和 AAEC 事件语义。
- `crates/open-edifier`：高层 facade 与驱动选择。
- `crates/open-edifier-cli`：面向用户和本地 Agent 的 CLI。
- `apps`：相互平行的最终用户产品；macOS 已交付 MVP，iOS 尚未交付。
- `bindings`：平台产品消费 SDK 的语言桥接，不是独立控制面。
- `integrations`：Home Assistant 等平行集成，不依赖其他应用产品。
- `docs`：已经确认的架构与协议事实。
- `research`：只存放脱敏、可复现的研究说明，不存原始专有产物。

## 架构约束

- Rust SDK 是唯一共享能力层。CLI、iOS、macOS 和 Home Assistant 必须保持平行，不能互相成为运行时前置依赖。
- CLI 和本地 Agent 直接使用 Rust SDK/CLI；macOS 通过最小 C ABI 静态链接 SDK；iOS 和 Home Assistant 使用规划中的对应绑定。
- 公共纯前端 WebUI 已归档。`docs/archive/` 中的内容是历史研究记录，不是活跃产品要求。
- S260 使用两条独立路径：framed JSON 负责完整状态和经过确认的修改；`BB EC` 负责低延迟实时事件。
- 应用和绑定必须使用公共状态、命令和事件类型，不得复制 wire protocol。
- 控制端口由型号驱动提供；发现层只返回设备身份与网络地址。
- `open-edifier-aaec` 只共享帧结构。输入源子命令、命令编号和载荷含义必须留在经过验证的型号驱动中。
- 保持同步核心，不要仅为未来应用引入 async runtime。平台层可自行放到后台线程或 actor 中。
- 第二个型号出现真实差异前，不要创建复杂插件系统、通用传输框架或推测性抽象。
- 未知但校验有效的事件应保持可观察，不能静默丢弃。

## S260 已验证边界

- TCP 控制端口为 `8080`。
- JSON 请求是无帧头的紧凑 UTF-8；响应使用 `EE DD FF EE` 和两字节大端长度。
- 当前 S260 传输为明文，不要重新暴露未经验证的 XOR 配置。
- JSON 输入源编号为蓝牙 0、AUX 1、USB 2、AirPlay 3。
- AAEC 输入源编号为蓝牙 1、AUX 2、USB 3、AirPlay 4，S260 子命令为 `0x1E`。
- 输入源、音量和 EQ 修改必须在 ACK 后查询并验证状态。
- 播放命令只能保证设备确认接收，实际效果取决于输入源。
- 所选输入源不可用时，设备可能在验证成功后再次回退；持续状态以事件流或刷新结果为准。
- 恢复出厂、固件升级、关机、重命名等破坏性能力不得进入公开 API，除非先有书面安全设计和明确需求。

## 状态与隐私

- 厂商完整状态响应可能包含 Wi-Fi、MAC、蓝牙配对记录等私有信息。
- 使用私有、强类型 Serde wire model 解析协议，只向公共状态投影稳定且必要的字段。
- 不得重新添加公开 `raw` 状态字段，也不得默认记录完整响应。
- 禁止提交真实 IP、MAC、主机名、Wi-Fi 名称、配对设备、凭据或本地绝对路径。
- 禁止提交厂商 APK、DEX、固件、手册、解包库、原始抓包或其他专有二进制。
- 测试 fixture 必须是合成或脱敏数据；即使是假数据，也优先使用明显不可误认成真实设备的信息。

## 新型号支持规则

声明支持新型号前，必须具备以下证据：

1. 明确的硬件型号和固件版本。
2. 脱敏 fixture、确定性 mock 或实机验证记录。
3. 每个公开命令的写后验证或清晰说明为什么只能确认 ACK。
4. 独立记录发现方式、默认控制端口、源编号和型号差异。
5. 未知字段、命令和错误响应的安全处理方式。

公开同族项目只能用于提出候选假设，不能替代本型号验证。不要因为端口、magic 或命令编号相同就声明兼容。

## 实机测试规则

- 测试前先读取并记录输入源、音量、EQ 和播放状态。
- 优先执行只读检查和 mock；只有功能验证需要时才修改实体设备。
- 修改测试必须使用小幅、安全值，并在结束时恢复音量、EQ 和可用的输入源。
- USB 已断开或输入源不可用时，不要反复强制切换；记录设备自动回退行为即可。
- 不执行恢复出厂、固件升级、关机或其他不可逆命令。
- 不通过重启路由器、断开用户网络或暴露端口来测试重连；使用本地双连接 mock。
- 最终交付说明实机验证了什么、没有验证什么，以及设备最终状态。

## 代码与 API 规则

- 保持公共 API 小而明确，优先表达已经验证的能力。
- 对网络、JSON、长度、数值转换和设备响应执行严格校验；响应始终按不可信输入处理。
- 状态修改成功不能只依赖 ACK；设备支持查询时必须读取并比较最终状态。
- 事件 decoder 必须支持分片、同包多帧、前导噪声、坏帧恢复和心跳过滤。
- 长期事件流应保持可取消，并在已建立连接意外关闭后使用有界退避重连。
- 不隐藏错误上下文，不把协议拒绝统一压成普通字符串成功输出。
- 新依赖必须有当前需求；标准库或现有依赖足够时不要增加 crate。
- 不为消除少量重复引入难以审查的泛型或宏。

## 文档规则

- 仓库内面向用户的 Markdown 使用中文。
- 命令、JSON 字段、协议 magic、Rust 类型名和项目名称保持原文。
- 已交付能力与路线图必须明确分开，禁止把规划中的应用或绑定写成已完成。
- 不得从归档文档推导新的产品工作；若要恢复 WebUI，必须先更新活跃架构和路线图。
- 协议文档只记录已确认行为；探索性假设放在研究说明中，并标明验证状态。
- 更新行为、API 或安全边界时，同步修改 README、对应协议/架构文档和 `CHANGELOG.md`。
- 外部公开文档不得出现本地绝对路径或个人设备信息。

## 修改工作流

- 先阅读当前实现、测试和相关文档，再决定修改范围。
- 工作区可能包含用户改动；保留无关变更，只修改任务范围内文件。
- 使用 `rg` 搜索文本和文件。
- 使用 `apply_patch` 修改文件，不用 shell 重写文件内容。
- 未经用户明确要求，不创建 commit、不 push、不发布 crate、不创建 GitHub 仓库或 PR。
- 不使用破坏性 Git 命令清理工作区。

## 质量门

代码修改完成后运行：

```bash
cargo fmt --all -- --check
cargo clippy --workspace --all-targets --locked -- -D warnings
cargo test --workspace --all-targets --locked
RUSTDOCFLAGS="-D warnings" cargo doc --workspace --no-deps --locked
cargo package --workspace --allow-dirty --no-verify
```

内部 crate 尚未发布到 crates.io 时，package 检查使用 `--no-verify`。发布顺序为 core、AAEC、S260、discovery、facade、CLI。

仅修改文档时，至少检查：

- Markdown 相对链接仍然存在。
- 没有完整英文叙述残留，技术标识除外。
- 没有真实设备信息、本地绝对路径或专有产物。

交付时报告实际运行的验证，不要把未执行的远端 CI 或实机场景描述为已经通过。
