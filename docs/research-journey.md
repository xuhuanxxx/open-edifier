# 从一条心跳，到局域网控制

这是一份经过脱敏的 EDIFIER S260 支持发现过程，既记录成功，也保留失败的假设，因为那些弯路解释了当前驱动中的许多设计选择。

本仓库不包含厂商应用二进制、固件、设备标识或私有网络抓包。其他 EDIFIER 型号的公开研究为我们提供了假设；所有被记录为 S260 支持的行为，都在固件 `01.00.00` 的 S260 实机上经过了验证。

## 最初目标

需求其实很小：不拿遥控器就能在蓝牙、AUX、USB 和 AirPlay 之间切换，再把同一套控制面开放给 CLI 和未来的本地集成。

USB 是最直观的候选，但最终证明它不适合作为控制面。音箱通过 USB 暴露 Audio 和 HID Consumer Control 接口，而主机音量与音箱内部音量相互独立。音箱已经为了 AirPlay 接入局域网，因此网络路径更有希望。

## 找到控制入口

S260 通过 mDNS 广播 AirPlay 服务。广播可以识别设备并解析主机地址，但它公布的媒体端口并不是控制端点。一次小范围端口探测找到了第二个 TCP listener：`8080`。

这里不讲 HTTP。保持一条空闲的裸 TCP 连接，反而会收到以 `BB EC` 开头的短二进制帧。载荷中包含可识别的设备身份字节，最后一个字节又恰好等于前面所有字节的 wrapping sum。这说明流量具有结构，而不是随机乱码；但它是否属于请求/响应协议，当时仍未确定。

## 命令扫描带来的假阳性

早期 probe 把 `BB EC` 当作请求前缀，并扫描候选命令。看起来很多命令都会响应，但重放完全相同的请求时，结果不同，甚至没有结果。

真正有用的实验是停止发送。在空闲连接上，同一条帧每隔数秒就会到达；实体状态变化时还会出现其他帧。此前命令扫描中绝大多数所谓的“响应”，只是恰好撞上了未经请求的心跳或状态事件。

研究方法因此发生了变化：

1. 在把流量归因给请求之前，先记录空闲基线。
2. 候选命令必须能够重复产生确定性行为。
3. 把切换输入源等实体操作视作带标签的事件。
4. 每次修改都通过查询或观察结果状态进行验证。

这个区别在生产实现中仍然重要：同一个 TCP 端口上，`BB EC` 流量可能与 framed JSON 交错出现。

## 识别协议族

网络实验继续进行的同时，另一条并行研究任务搜索了相邻 EDIFIER 产品的公开逆向项目。关键的不对称关系终于出现：同族协议的请求以 `AA EC` 开头，响应和事件才以 `BB EC` 开头。公开研究还提供了输入源、音量、播放和 EQ 的候选命令编号。

输入源查询随后变得可以稳定复现：

```text
请求  AA EC 61 00 00 F7
响应  BB EC 61 00 02 1E 03 2B
```

响应报告输入源 `3`；经过验证的 S260 映射表将它解释为 USB。之后的实验继续确认了音量读写、输入源切换、播放操作和 EQ 预设。

同族并不意味着完全相同。例如，S260 输入源载荷使用子命令 `0x1E`，在其他型号上观察到的另一个值则无法工作。早期解析还一度把长度字段当作两字节；精确的帧边界最终证明，OpenEdifier 采用的 S260 表达方式是两字节小端命令，后接一字节载荷长度。

因此，OpenEdifier 只在 `open-edifier-aaec` 中共享二进制分帧；命令语义仍由每个型号驱动独立拥有和验证。

## 同一端口上的第二套协议

并行研究还对 EDIFIER HOME Android 应用进行了本地静态字符串检查。结合公开的 ES300 研究，它提示 `8080` 端口可能也接受 JSON。本仓库不保留应用二进制或任何解包产物。

发送一条紧凑、无帧头的状态请求后成功收到了响应：

```json
{"id":"<unique-id>","payload":"status_query"}
```

响应以 `EE DD FF EE` 开头，后接两字节大端长度和自描述 JSON。它暴露的状态远比二进制查询路径丰富：输入源、内部音量范围、EQ 预设、播放能力、固件信息，以及 OpenEdifier 有意不提供控制的其他功能。

因此，第一版控制台使用 JSON 发送命令和查询状态。由于 JSON 路径不会向空闲 JSON 客户端广播状态变化，当时的事件命令只能轮询状态。

后续测试把这个限制与最初观察到的二进制现象统一起来：JSON 控制面负责请求/响应，独立的 `BB EC` 通道负责推送实时状态事件。当前 S260 驱动因此同时使用两者：

```text
framed JSON  -> 完整状态与经过验证的修改
BB EC 事件   -> 低延迟输入源、音量、播放和 EQ 更新
```

这种组合比强迫任何一套协议承担全部职责更可靠。观察到的固件可能丢失二进制命令响应，或者只回答连接上的第一条命令；JSON 命令则会得到确认，还能通过完整状态查询进行后续验证。

## 为什么推进得这么快

这是一场 Agent 辅助的硬件实验，而不是一连串孤立的猜测。一条工作流持续测试实时 TCP 服务，另一条工作流搜索先行研究并检查应用字符串。人类操作者则通过插拔 USB、按下实体控制键、确认输入源指示灯变化，为网络观察提供不可替代的实体标签。

这种分工很重要：公开研究缩小候选协议范围，自动化让重复 probe 的成本足够低，而实体设备负责裁定哪种解释才是真的。

## 实体实验提供了什么

音箱本身才是最终事实源。只有把网络帧与可观察操作关联起来，字节才真正具有意义：

- 输入源命令改变了前面板指示灯；
- 音量写入后，从音箱内部 `0-30` 范围中读回确认；
- EQ 修改后重新查询，并在测试结束后恢复；
- 播放命令得到确认，但效果取决于输入源；
- 实体控制产生了带标签的二进制状态事件；
- 主机侧 USB 音量变化不会改变音箱内部音量。

这些检查避免把一条被接受的数据包或无关心跳误判为真正有效的控制。

## 带入 SDK 的经验

- 面对未知长连接，先听再说。
- 校验和能验证帧假设，不能证明命令语义。
- 同族协议研究适合提供候选，不足以作为型号支持证据。
- 设备允许时，修改状态的 API 应验证最终结果。
- 校验有效但未知的事件应保持可观察，而不是直接丢弃。
- 从状态中发现破坏性能力，不代表它应该默认进入公开 API。
- 原始研究产物和个人设备数据必须留在公开仓库之外。

最终实现有意比研究过程平淡：小而安全的 API、确定性 mock 测试、带类型的事件，以及允许其他型号与 S260 存在差异的驱动边界。

OpenEdifier 不会照搬这些项目的兼容性声明。它们帮助我们缩小假设范围；本文描述的 S260 行为则由实机独立验证。
